# Python 变更：文档同步、风格与测试

## 适用场景

在根据用户需求**修改、优化或扩展**代码时，按本技能执行，确保：`docs/`（及同类说明文档）与**真实行为**一致，代码符合 **Python 风格**，且对**模块级改动**补充**单元测试**。

## 工作流（按顺序）

### 1. 理解需求与影响面

- 明确用户要改的行为、接口、配置或边界情况。
- 识别将触及的模块、公共 API、CLI/配置项及已有文档中的描述。

### 2. 实现代码变更

- **最小必要改动**：只改完成任务所需的代码，避免无关重构。
- **风格**：遵循 [PEP 8](https://peps.python.org/pep-0008/)；若仓库已有格式化/检查工具（如 `ruff`、`black`、`flake8`），以项目配置为准并与之对齐。
- **类型与命名**：与周边代码一致；新增公共接口宜有清晰命名与必要类型标注（若项目已采用类型提示）。
- **行为完整性**：实现后自检边界条件、错误路径与向后兼容性（若适用）。

### 3. 文档与代码一致性（强制）

当本次改动使行为、默认值、配置字段、观测空间/动作空间、奖励、环境重置逻辑等与文档**不一致**时：

- **必须**更新对应文档（通常为仓库内 `docs/` 下的 Markdown，或其它用户指定的说明文件）。
- 更新原则：
  - 描述**当前代码的实际行为**，而非计划或未实现的特性。
  - 若接口变更，同步：参数名、类型、默认值、示例命令或配置片段。
  - 避免「文档写一套、代码跑另一套」；若暂时无法实现文档所述行为，应改文档以反映现状，或在与用户确认后改代码达成文档（二选一，禁止长期漂移）。

### 4. 单元测试（模块改动时强制）

对**本次修改所涉及的 Python 模块**（新建或实质性改动逻辑时）：

- **新增或扩展**面向该逻辑的单元测试（项目常用 `pytest` 则使用 `pytest`）。
- 测试应覆盖：
  - 核心正确路径；
  - 本次修复或新增相关的边界与回归点；
  - 可在不依赖重量级外部环境（如真实仿真器）时，优先用 mock/fixture 隔离。
- 放置位置与命名遵循仓库既有测试目录结构（如 `tests/`）。

### 5. 自检

- 运行项目推荐的静态检查/格式化（若存在）。
- 运行与改动相关的测试子集或完整测试套件（按项目惯例）。
- 再次核对：文档段落是否与当前实现逐条对应。

## 交付说明（对用户）

简要说明：

1. **代码做了什么**（行为层面）。
2. **文档更新了哪些文件**（若有），以及为何。
3. **新增/修改了哪些测试**，如何运行。

## 反模式（避免）

- 只改代码不读文档，导致 `docs/` 过时。
- 为「省事」在文档中模糊表述，掩盖确定的行为。
- 模块逻辑已变却不加测试，或只手动跑一次脚本代替自动化测试。
- 风格与周边文件严重不一致（除非用户明确要求统一重构）。

## 何时向用户确认

- 文档与代码冲突且产品意图不明确（应改代码还是改文档）。
- 缺少测试基础设施或需引入新依赖才能测试。
- 变更会破坏公开 API 且用户未声明兼容性策略。
