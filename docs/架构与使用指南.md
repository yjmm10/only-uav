# onlyuav 架构与使用指南

本文档描述仓库的工程结构、运行流程、Hydra 配置体系、训练与评估产物、技术方案要点，以及可扩展方式。内容与当前源码保持一致，便于复现、写论文附录或二次开发时查阅。

**相关文件**：需求与顶层设计见同目录下的《规划文档.md》；**环境与各模块默认行为、处理流程及示例**见《环境与模块说明.md》；日常命令速查见仓库根目录 `README.md`。

---

## 1. 项目定位

**onlyuav** 是一个**模块化**的无人机任务卸载仿真环境，对外暴露标准 **`gymnasium.Env`** 接口，便于接入各类强化学习库。

- **仿真**：移动、无线信道、功耗、任务到达、本地/边缘计算、电池与奖励等子模块通过**注册表**按配置组装。
- **配置**：使用 **Hydra + OmegaConf** 管理多层 YAML，可在命令行覆盖任意叶子字段。
- **训练**：入口统一为 `onlyuav.train`；根据 `experiment.train.backend` 分发到 **Stable-Baselines3（默认）**、**Ray RLlib**、**Tianshou**、**DI-engine** 等实现。
- **评估**：当前 `onlyuav.eval` **仅支持 SB3** 导出的 `.zip` 策略；其它后端需在对应库内自行评估或扩展 `eval.py`。

**依赖与版本**：见根目录 `pyproject.toml`（Python ≥ 3.10；核心依赖含 `gymnasium`、`hydra-core`、`omegaconf`、`numpy`、`stable-baselines3` 等）。

## 3. 核心架构

### 3.1 组件注册表与环境构建

**文件**：`onlyuav/core/env_builder.py`

- **`ComponentRegistry`**：类属性 `_registry` 保存 `名称字符串 → 类` 的映射。
  - **`@ComponentRegistry.register("SomeName")`**：装饰器将实现类注册到 `_registry`。
  - **`instantiate(cfg)`**：读取字典 `cfg["type"]` 作为注册名，`cfg.get("params", {})` 作为关键字参数实例化。
- **`EnvBuilder.build(module_cfg)`**：
  - 将 `DictConfig` 转为普通 `dict`（`resolve=True`）。
  - 遍历除 `env_params`、`meta` 外的每个键；若子配置是包含 `"type"` 的字典，则调用 `instantiate` 得到模块实例。
  - 固定组装 **`DroneEnv`**，并传入：`mobility`、`channel`、`power`、`task`（代码中为 `task_gen`）、`computing`、`energy`、`reward`、`observation`、`action_interpreter`，以及 `env_params` 中的 `server_pos`、`max_steps`、`dt`。

因此：**只要 YAML 中的模块键名与 `EnvBuilder` 期望一致，且 `type` 已注册**，即可替换实现而无需改 `DroneEnv` 的 import 列表（但若增加**全新键**的模块，需同步修改 `EnvBuilder` 与 `DroneEnv`）。

### 3.2 抽象接口

**文件**：`onlyuav/core/interfaces.py`（继承 `BaseComponent`）

| 接口 | 主要方法 | 语义 |
|------|----------|------|
| `IMobility` | `step`, `state` | 根据控制量更新位置/速度，返回状态快照 |
| `IChannel` | `rate(tx_pos, rx_pos)` | 返回信道速率（bps） |
| `IPower` | `compute(velocity, computation_load)` | 返回总功耗 |
| `ITaskGenerator` | `sample(current_time)` | 返回当前时间步到达的任务列表 |
| `IComputing` | `process(task, offload_target, channel_rate)` | 返回执行时间、能耗、是否成功等 |
| `IEnergy` | `consume`, `remaining` | 消耗能量、查询剩余电量 |
| `IReward` | `compute(state: dict)` | 根据本步统计量返回标量奖励 |
| `IObservation` | `get_obs(env_state: dict)` | 将环境状态映射为观测向量 |
| `IActionInterpreter` | `interpret(raw_action)` | 将 RL 原始动作映射为移动指令、卸载目标、CPU 频率等 |

### 3.3 默认组件加载

**文件**：`onlyuav/models/__init__.py` 中的 **`load_default_components()`**

通过显式 `import onlyuav.models.<子包>` 触发各模块文件顶层的注册装饰器执行。训练与评估入口在 `main` 开始处会调用该函数，**保证**在 `EnvBuilder.build` 之前注册表已就绪。

### 3.4 环境本体 `DroneEnv`

**文件**：`onlyuav/envs/drone_env.py`

- **动作空间**：`gymnasium.spaces.Box`，形状与边界为  
  `low=[-1,-1,0]`, `high=[1,1,1]`，dtype `float32`（二维平面移动 + 卸载连续 gate；机载 CPU 频率为配置常量，见 `StandardInterpreter.fixed_local_cpu_freq`）。
- **观测空间**：`Box`，`shape=(8,)`，`low=-inf`, `high=inf`，`float32`。具体语义由 `obs_model` 决定。
- **`reset`**：重置步数、任务队列、累计量；调用 `mobility`、`energy`、`task_gen`、`action_interp`、`reward` 的 `reset`；返回观测与空 `info`。
- **`step` 逻辑顺序**（与代码一致）：
  1. `action_interp.interpret(action)` → `move_cmd`, `offload_target`, `fixed_compute_for_power`（默认实现下为固定机载频率，供 `power.compute`）
  2. `mobility.step(move_cmd)` → `pos`, `vel`
  3. `channel.rate(pos, server_pos)` → `rate`
  4. `task_gen.sample(self.steps)` 到达任务并入队
  5. 若队列非空：弹出队首任务，`computing.process(task, offload_target, rate)`；根据 `success` 与 `exec_time <= task["max_delay"]` 判定成败；更新 `delay`、`comp_energy`、`completed`/`failed`、`total_completed`
  6. `power.compute(vel, fixed_compute_for_power)`，`energy_model.consume(power_total, dt=self.dt)`，累计 `total_delay`、`total_energy`
  7. `reward_model.compute({completed, energy_cost, delay})`
  8. `steps += 1`；`terminated = remaining_energy <= 0`；`truncated = steps >= max_steps`
  9. 组装 `info`（含 `distance`、`remaining_energy`、`completed_tasks` 等），`_get_obs()` 返回

**`_get_obs`**：从 `mobility.state()` 与电量、队列长度、`server_pos` 构造 `env_state`，交给 `obs_model.get_obs`。

### 3.5 RL 公共工具

**文件**：`onlyuav/rl/common.py`

- **`build_env(cfg, fixed_reset_seed)`**：`load_default_components()` → `EnvBuilder.build(cfg.modules)`；若 `fixed_reset_seed` 非空，外包一层 **`FixedSeedResetWrapper`**：每次 `reset` 强制传入该 `seed`，便于固定任务流复现。
- **`CONFIG_DIR`**：指向仓库 `configs/` 目录，供超参 YAML 查找。
- **`to_jsonable`**：将 numpy 标量/数组等转为可 JSON 序列化的 Python 类型。
- **`append_jsonl`**：向 JSONL 文件追加一行（部分后端可能使用）。

## 4. 处理流程

### 4.1 训练主流程

1. **`python -m onlyuav.train`** 或 **`onlyuav-train`** → `onlyuav/train.py` 的 `main`。
2. **`load_default_components()`**。
3. **`onlyuav.rl.runner.run_training(cfg)`**：
   - **算法列表**：若配置 `experiment.train.algos` 为非空列表，则逐项训练；否则取 **`resolve_training_algorithm_name`** 的单个算法（见下文）。
   - **`multi_mode`**：`len(algo_list) > 1`。
   - 读取 `total_timesteps`、`fixed_reset_seed`、`resume`、`parallel_multi_algo`、指标相关开关、`backend` 等。
   - **`parallel_multi_algo` 且 `multi_mode`**：为每个算法构造子进程命令  
     `python -m onlyuav.train`，并传入当前 Hydra 的 `modules`、`experiment` 选择，关闭 `algos`、单算法 `algorithm=<algo>`，关闭并行多算法标志；等待所有子进程退出码，若有非零则 `SystemExit`。
   - 否则对每个算法调用 **`train_one`** → 按 `backend` 调用 `train_sb3` / `train_rllib` / `train_tianshou` / `train_di_engine`。
4. **SB3 路径**（`onlyuav/rl/sb3_train.py`）概要：
   - `get_algo_class` + `load_algo_hparams` → 弹出 `policy`，构造算法实例。
   - `build_env(cfg, fixed_reset_seed)`。
   - 若 `resume` 为真且存在 checkpoint：从磁盘 **load** 并续训；否则 **新建** 模型。
   - `model.learn(..., callback=TrainingCallback)`：回调中按步聚合 SB3 logger 键、可选写入 `metric_jsonl`。
   - **`model.save(model_path)`**；写入 `log_json`（episode 回报与时间步列表）。

### 4.2 算法名解析（单算法模式）

**文件**：`onlyuav/rl/hparams.py` 中的 **`resolve_training_algorithm_name`**

优先级：

1. `experiment.train.algorithm`（非空且非未解析的 `${...}` 插值）
2. 兼容旧键 `experiment.train.algo`
3. 默认 **`ppo`**

### 4.3 后端与算法白名单

**文件**：`onlyuav/rl/registry.py`

- **`normalize_backend`**：去空白、小写、`-` → `_`；别名 `di`、`ding` → `di_engine`。
- **`SUPPORTED_ALGOS`**（当前实现）：

| backend | 支持的 algorithm 名称 |
|-----------|------------------------|
| `sb3` | `ppo`, `a2c`, `sac`, `td3` |
| `rllib` | `ppo`, `sac` |
| `tianshou` | `ppo`, `sac` |
| `di_engine` | `ppo` |

不支持的组合会在训练开始时 **`assert_algo_supported`** 报错。

### 4.4 超参合并顺序

**文件**：`onlyuav/rl/hparams.py` 的 **`load_algo_hparams(cfg, algo_name)`**

对给定 **`algo_name`**（小写键，如训练循环中的 `ppo`）：

1. **`builtin_defaults.get_builtin(backend, key)`**（`onlyuav/rl/builtin_defaults.py`）：与该库 API 对齐的默认字典。
2. **`experiment.train.shared_algo_kwargs`**：合并覆盖（适合多算法共用的 `learning_rate`、`gamma` 等）。
3. 若存在文件 **`configs/<BACKEND_PARAM_DIR[backend]>/<algo_name>.yaml`**，则加载并合并。目录映射为：  
   `sb3` → `algorithm/`；`rllib` → `algorithm_rllib/`；`tianshou` → `algorithm_tianshou/`；`di_engine` → `algorithm_ding/`。
4. **仅当** `backend == "sb3"` **且** `algo_name` 与 **`resolve_training_algorithm_name(cfg)`** 一致时：若 Hydra 根上存在 **`cfg.algorithm`** 组（需在 `config.yaml` 的 `defaults` 中加入如 `- algorithm: ppo`），则将该组内容合并进来。
5. **`experiment.train.algo_kwargs`**：最高优先级覆盖。

内置 SB3 默认键示例（节选）：PPO 含 `n_steps`、`batch_size`、`gae_lambda` 等；SAC 含 `buffer_size`、`tau` 等；TD3 含 `policy_delay` 等。完整表以 `builtin_defaults.py` 为准。

### 4.5 评估主流程

**文件**：`onlyuav/eval.py`

1. `load_default_components()`；**强制**能导入 `stable_baselines3`。
2. **`normalize_backend`**；若不为 `sb3`，**退出**并提示仅支持 SB3 `.zip`。
3. **评估算法列表**：`experiment.eval.algos` → 否则 `experiment.eval.algo` → 否则 `experiment.train.algos` → 否则 **`resolve_training_algorithm_name`**。
4. 对每个算法：`get_algo_class` → 解析 **`model_path`** 与 **`output_json`**（见下一节）→ `EnvBuilder.build(cfg.modules)` → `algo_class.load` → **`evaluate_policy`**（`onlyuav/evaluation/evaluator.py`）→ 将 summary 中 tuple 形式的 `(mean, std)` 转为嵌套 dict 写入 JSON。

**评估循环**：每 episode `reset` 后反复 `policy_fn(obs)` → `env.step`，直到 `terminated or truncated`；指标由 **`MetricsTracker`** 累计。

**汇总指标**（`onlyuav/evaluation/metrics.py` 的 `get_summary`）：跨 episode 对以下量做均值与标准差（返回为 `(mean, std)` 元组，eval 写入 JSON 时展开）：

- `avg_reward`, `avg_completed_tasks`, `avg_delay`, `avg_energy`, `avg_offload_rate`, `avg_distance`  
以及 `num_episodes`。

**画图**：`onlyuav/plot.py` 中 `plot_eval_summary` 默认绘制 `avg_reward`、`avg_completed_tasks`、`avg_delay`、`avg_energy`、`avg_offload_rate`（若 JSON 中缺某键则跳过）。

## 5. 路径解析与输出文件

### 5.1 输出路径 `_resolve_output_path`

**共同规则**：若路径字符串中含 **`{algo}`**，则格式化为当前算法小写名（例如 `trained_models/{algo}_edge` → `trained_models/ppo_edge`）。默认实验配置中的 `model_path`、`log_json` 等通常已带 **`{algo}`**，多算法并行时各进程互不覆盖。

**多算法且路径中不含 `{algo}` 时的差异（与源码一致）**：

- **训练**（`onlyuav/rl/runner.py`）：`Path.with_name(f"{base.name}_{algo}")`，即在**完整文件名**（含后缀）后拼接 **`_<algo>`**。例如 `logs/out.json` 可能变为 `logs/out.json_ppo`。若需可读性更强的 `out_ppo.json`，请在配置中显式使用 **`{algo}`** 占位或自定义命名规则。
- **评估**（`onlyuav/eval.py`）：`Path.with_name(f"{base.stem}_{algo}{base.suffix}")`，即在**主文件名与后缀之间**插入 **`_<algo>`**，例如 `eval_metrics.json` → `eval_metrics_ppo.json`。

**单算法**：路径按上述规则解析后原样使用（或仅做 `{algo}` 替换）。

### 5.2 指标 JSONL 路径 `_resolve_metric_path`

若 `metric_jsonl` 为**相对路径**，则写在 **Hydra 当前 run 的 `output_dir`** 下（`HydraConfig.get().runtime.output_dir`），便于每次实验隔离；绝对路径则直接使用。

### 5.3 SB3 模型保存与续训

- **`model.save(str(model_path))`**：Stable-Baselines3 通常写入 **`*.zip`**（若路径无后缀，以控制台打印为准）。
- **续训**：在 `resume=True` 时查找 `model_path` 或 `model_path.with_suffix(".zip")`；存在则 `load` 后继续 `learn`；日志中会打印 `RESUME` 或 `FRESH_START`。

### 5.4 训练产物摘要

| 产物 | 配置键（默认 experiment） | 说明 |
|------|---------------------------|------|
| 策略权重 | `train.model_path` | SB3 多为 `.zip` |
| Episode 级 JSON | `train.log_json` | 含 `timesteps`、`episode_rewards` 等，供 `plot` 学习曲线 |
| 迭代指标 JSONL | `train.metric_jsonl` | 每行一条 JSON，含 SB3 logger 抽取字段、可选时间戳与 step_reward |
| Hydra 运行日志 | Hydra 自动生成 | 如 `outputs/<日期>/<时间>/train.log` |

### 5.5 评估产物

| 产物 | 配置键 | 说明 |
|------|--------|------|
| 指标 JSON | `eval.output_json` | 多算法时自动加 `_<algo>` 后缀（与 `model_path` 规则类似） |

## 6. Hydra 配置教程

### 6.1 总入口 `configs/config.yaml`

- **`defaults`**：列出要组合的配置组；当前典型为  
  `- modules: default`、`- experiment: default`、`- _self_`。
- 顶层常用键：**`seed`**、**`output_dir`**（Hydra 工作目录相关行为以 Hydra 版本为准）。

切换实验预设：

```bash
uv run python -m onlyuav.train experiment=local_only
```

只替换某一环境子模块：

```bash
uv run python -m onlyuav.train modules/computing=local_only
```

### 6.2 模块层 `configs/modules/default.yaml`

通过 **`defaults`** 引用各子 YAML，例如：

- `mobility: simple_point_mass`
- `channel: free_space`
- `power: simple_power`
- `task: poisson_arrival`
- `computing: edge_offloading` 或 `local_only`
- `energy: finite_battery`
- `reward: weighted_sum`
- `observation: full_obs`
- `action_interpreter: standard`
- `env_params: default`

每个子文件体一般为：

```yaml
type: RegisteredClassName
params:
  key: value
```

`type` 必须与 **`@ComponentRegistry.register`** 中的字符串**完全一致**（区分大小写）。

### 6.3 环境标量 `configs/modules/env_params/default.yaml`（示例）

- **`server_pos`**：边缘服务器位置，三维列表。
- **`max_steps`**：每回合最大步数，达到则 `truncated`。
- **`dt`**：物理步长，传入 `DroneEnv` 与部分模块。

### 6.4 实验层 `configs/experiment/default.yaml` 字段说明

**`train` 段**

| 字段 | 含义 |
|------|------|
| `backend` | `sb3` / `rllib` / `tianshou` / `di_engine` |
| `algorithm` | 单算法训练时的算法名 |
| `algos` | 非空列表时启用多算法；`null` 表示单算法 |
| `total_timesteps` | 训练总步数（SB3 传给 `learn`） |
| `shared_algo_kwargs` | 多算法共用超参 |
| `algo_kwargs` | 最高优先级超参覆盖 |
| `rllib` | RLlib 专用：`num_env_runners`、`num_cpus`、`use_new_api_stack` 等 |
| `parallel_multi_algo` | 多算法时是否子进程并行 |
| `resume` | 是否尝试从已有模型续训 |
| `fixed_reset_seed` | 固定每次 reset 的 seed；`null` 表示不包 Wrapper |
| `model_path` / `log_json` / `metric_jsonl` | 输出路径，支持 `{algo}` 占位 |
| `metrics_log_every_steps` | JSONL 写入间隔（步） |
| `metrics_include_step_reward` | 是否在 JSONL 中记录逐步奖励 |
| `metrics_include_timestamp` | 是否在 JSONL 中记录 UTC 时间戳 |

**`eval` 段**

| 字段 | 含义 |
|------|------|
| `algo` / `algos` | 指定评估算法；可留空以继承训练列表或默认单算法 |
| `episodes` | 评估 episode 数 |
| `output_json` | 指标输出路径，支持 `{algo}` |

**`plot` 段**：默认输出文件名占位，供文档或脚本引用。

### 6.5 与 `default` 差异化的实验预设（示例）

| 文件 | 典型差异 |
|------|-----------|
| `local_only.yaml` | `computing` 为本地；`model_path` / `log_json` / `eval.output_json` 带 `local` 后缀区分 |
| `high_energy_long_train.yaml` | `total_timesteps` 更大、`episodes` 更多、输出文件名带 `high_energy` |

实际以各 YAML 为准。

### 6.6 使用 Hydra 的 `algorithm` 组覆盖 SB3

若希望在配置树中出现 **`algorithm.xxx`** 形式，需在 **`configs/config.yaml`** 的 **`defaults`** 中增加一行，例如 `- algorithm: ppo`，并提供对应 `configs/algorithm/ppo.yaml` 或与 Hydra 搜索路径一致的组。合并优先级见 **§4.4**；**`algo_kwargs` 仍为最高优先级**。

## 8. 技术方案归纳

1. **配置驱动组装**：环境由数据定义，新增同类模块优先通过 YAML 切换 `type`/`params`。
2. **注册表模式**：避免在中央维护巨大的 `if-elif` 导入表；新类自注册，导入链由 `load_default_components` 统一触发。
3. **接口分层**：`interfaces.py` 约束子模块输入输出，便于单测与替换。
4. **训练与仿真解耦**：仿真仅依赖 `gymnasium`；各 RL 库在 `rl/` 下隔离，统一由 `runner` 分发。
5. **超参分源合并**：内置默认 + 共享 + 可选 YAML + Hydra `algorithm` 组 + `algo_kwargs`，兼顾「只改算法名」与「论文级微调」。
6. **可复现性**：可选固定 reset seed；全局种子配置；Hydra 输出目录隔离指标。

## 9. 如何扩展（实施检查清单）

### 9.1 新增一种环境子模块（如新型信道）

1. 在 `onlyuav/models/channel/`（或对应子包）实现类，继承 **`IChannel`**，实现 `rate` 等方法。
2. 使用 **`@ComponentRegistry.register("YourChannelName")`** 注册（名称与 YAML `type` 一致）。
3. 在 **`onlyuav/models/channel/__init__.py`** 中 import 该模块，确保 `load_default_components()` 能加载到。
4. 新增 **`configs/modules/channel/your_channel.yaml`**，内容为 `type` + `params`。
5. 修改 **`configs/modules/default.yaml`**（或实验级 `defaults`）将 `channel` 指向新 yaml。
6. 若 **`EnvBuilder`** 当前未包含该键，需扩展 `build` 与 **`DroneEnv.__init__`** 及 `step` 逻辑（本仓库信道已存在键名 `channel` 时仅需换实现）。

### 9.2 修改观测或动作维度

1. 同步修改 **`DroneEnv`** 的 **`observation_space` / `action_space`**。
2. 修改 **`IActionInterpreter`** 与 **`IObservation`** 的实现及对应 YAML `params`。
3. 重新训练策略；旧 checkpoint **不兼容**新空间时需删除或换路径。

### 9.3 在 SB3 下新增算法名

1. 在 **`onlyuav/algorithms.py`** 的 `mapping` 增加库类与短名。
2. 在 **`onlyuav/rl/builtin_defaults.py`** 增加该算法的默认参数字典。
3. 在 **`onlyuav/rl/registry.py`** 的 **`SUPPORTED_ALGOS["sb3"]`** 中加入短名。
4. 可选：新增 **`configs/algorithm/<name>.yaml`** 微调。

### 9.4 新增 RL 后端

1. 实现 **`onlyuav/rl/<name>_train.py`**，`train_*` 函数签名对齐 `runner.train_one` 中 `common` 字典的用法（可参考 `train_sb3`）。
2. 在 **`onlyuav/rl/runner.py`** 的 `run_training` 中增加分支。
3. 更新 **`normalize_backend`**（若有别名）与 **`SUPPORTED_ALGOS`**。
4. 在 **`onlyuav/rl/hparams.py`** 的 **`BACKEND_PARAM_DIR`** 中注册配置子目录名；提供对应 `configs/algorithm_<backend>/` 下 YAML（可选）。
5. 在 **`pyproject.toml`** 增加 `optional-dependencies` 并在文档中说明安装方式。
6. **评估**：扩展 **`onlyuav/eval.py`** 的加载与推理逻辑，或保持仅在各库内评估。

### 9.5 新增实验预设

复制 **`configs/experiment/default.yaml`**，修改 `model_path`、`log_json`、`total_timesteps`、`modules` 引用等，通过 **`experiment=你的文件名`** 调用。

## 10. 常见问题（FAQ）

**Q：评估报错「仅支持 SB3」？**  
A：将 `experiment.train.backend` 设为 `sb3` 完成训练，或自行扩展 `eval.py` 以加载其它库格式。

**Q：多算法输出找不到模型？**  
A：优先在路径中使用 **`{algo}`** 占位符。若不使用，训练与评估对文件名的自动改写规则略有不同（见 **§5.1**）；评估 JSON 与训练日志请以各自 `_resolve_output_path` 行为为准，必要时用 `ls`/`find` 对照 Hydra 输出目录。

**Q：JSONL 不在项目根目录的 `logs/`？**  
A：相对路径的 `metric_jsonl` 会写到 **Hydra 的 `output_dir`** 下，属于预期行为。

**Q：切换算法后 loss 异常？**  
A：检查是否误用其它算法的专有键；使用 **`builtin_defaults`** 与 **`algo_kwargs`** 分层调试。
